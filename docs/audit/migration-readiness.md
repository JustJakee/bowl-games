# Migration Readiness

## CRA to Vite

### Current readiness

- Moderate. The frontend is small and has a single entry point, but it still relies on CRA conventions and has no automated tests protecting the move.
- Helpful factors:
  - No `react-router`.
  - No custom webpack config.
  - No environment variable reads in app source.
  - The app already uses modern React 18 and ESM-style imports.
- Drag factors:
  - CRA HTML template still uses `%PUBLIC_URL%`.
  - The app appears to deploy at `/leaderboard/`, which means Vite `base` handling must be decided explicitly.
  - There is stale CRA test scaffolding.
  - Multiple UI systems are loaded together, so CSS order regressions are possible.

### Main migration risks

- Environment variable changes:
  - None are used now, but any future build-time config will need to move from CRA `REACT_APP_*` conventions to Vite `VITE_*`.
  - Current Amplify config is file-based, so env migration is not the main blocker.
- Entry file changes:
  - `src/index.js` will become a Vite-style `main.jsx` or equivalent.
  - `ReactDOM.createRoot` logic itself is straightforward to carry over.
- `index.html` differences:
  - CRA placeholders like `%PUBLIC_URL%` will not work in Vite.
  - Asset references and manifest links must be rewritten for Vite semantics.
- Static asset path issues:
  - If production must remain under `/leaderboard/`, Vite `base` cannot be left implicit.
  - Any absolute asset assumptions need validation against the deployed subpath.
- Test changes:
  - The current test suite is effectively non-functional already, so a Vite migration should either replace the stale CRA sample test or consciously remove it in a follow-up.
- Amplify config import issues:
  - `amplifyconfiguration.json` import should remain workable, but it needs validation under Vite bundling and the intended deployment base path.

### Readiness judgment

- Technically ready once deployment-path decisions are documented.
- Operationally underprepared because there is no meaningful test coverage and deployment assumptions are not codified in repo config.

## Amplify Gen 1 to Gen 2

### Current readiness

- Moderate for resource count, low for production certainty.
- Helpful factors:
  - Only one backend category exists in repo: a single AppSync API.
  - No Cognito, S3, Lambda, REST API, or multi-resource dependency graph was found.
- Drag factors:
  - The current backend is generated by Amplify Gen 1 CLI conventions.
  - Frontend relies on generated GraphQL docs and committed `amplifyconfiguration.json`.
  - Production data expectations are unknown from the repo alone.

### Main migration risks

- Backend resource mapping:
  - The Gen 1 `@model` `Submission` schema will need an equivalent Gen 2 data definition.
  - Public API-key create/read behavior must be intentionally re-modeled rather than assumed.
- Data migration:
  - Existing submissions, if any, are stored in a DynamoDB table managed by Gen 1.
  - Picks are embedded as `AWSJSON` keyed by bowl display names, which is brittle for future schema evolution.
  - If historical picks matter, table export/import planning is required before cutover.
- Auth migration:
  - There is no Cognito resource yet, but the next phase intends to add auth.
  - Combining Gen 1 to Gen 2 migration with first-time auth introduction raises cutover risk if done in one PR.
- Generated client/config differences:
  - The app currently imports generated GraphQL operations and `amplifyconfiguration.json`.
  - Gen 2 will change how backend definitions, config generation, and client setup are produced.
- Deployment workflow:
  - Current repo does not encode the full deployment path.
  - Gen 2 will shift local and deployment workflows toward code-defined backend resources.
- Local dev workflow:
  - Current repo behavior is partially decoupled from AWS already because `AWS_DISABLED` is true.
  - Re-enabling real local backend flows will need explicit decisions about dev environments and seeded data.

### Readiness judgment

- The backend is small enough to migrate without a massive resource-conversion project.
- The main blockers are identity strategy, data preservation requirements, and missing deployment documentation.

## Authentication and user management

### Current readiness

- Low. The current app has no real account model.
- What exists now:
  - Hardcoded frontend gate in `Home.jsx`.
  - Freeform submission name and email stored per entry.
  - Public AppSync API-key access for reads and creates.
- What does not exist:
  - Cognito
  - session persistence
  - role-based authorization
  - ownership-based updates
  - account recovery or verified email flow

### Main migration risks

- Associating existing picks with future users:
  - Current records only have `name` and `email`.
  - There is no verified identity, no stable user ID, and no proof that a future Cognito user owns a historical submission.
  - Historical linkage will require a policy decision, not just code.
- Historical data sufficiency:
  - The repo suggests only `name`, `email`, picks JSON, and tie-breaker are stored.
  - That may be enough for a one-time claim flow if emails are real and unique, but the repo cannot prove that.
- Admin roles:
  - The future system will likely need at least one admin role for season setup, pick lock control, payout/export actions, and possibly score overrides.
- Pick editing and locking:
  - Current system treats picks as immutable after create.
  - Future editable-until-first-game behavior will require ownership, write authorization, and a clear locking rule based on kickoff time.
- Saving picks in progress:
  - No draft model exists now.
  - Drafts could be local-only, user-scoped in backend, or both. That choice affects schema design.
- Login persistence:
  - Current pseudo-login resets on refresh.
  - Real auth should define refresh persistence expectations explicitly.

### Readiness judgment

- Not ready for direct auth implementation without first deciding:
  - account identifier and sign-in method
  - how historical submissions map to accounts
  - admin role model
  - whether one account can own multiple entries

## Cross-cutting risks

- Sensitive config is committed:
  - `src/amplifyconfiguration.json` includes a live-looking AppSync API key.
- Public-read data exposure:
  - Submission emails are readable through `listSubmissions`.
- Data shape fragility:
  - Picks are stored as JSON keyed by bowl display name, not by durable game IDs.
- Deployment uncertainty:
  - `/leaderboard/` subpath behavior and hosting config are not codified in repo.
- Test gap:
  - Existing test file is stale and not useful as migration protection.

## Recommended migration order

1. Documentation and decision pass
   - Confirm production hosting path, data retention requirements, and account strategy.
2. Data model hardening
   - Define durable entities for users, entries, picks, games, lock state, and optional drafts.
3. Authentication design
   - Decide Cognito flows, admin roles, and historical-entry association policy.
4. Amplify Gen 1 to Gen 2 backend migration
   - Recreate the backend with the new data/auth model while keeping the frontend build system unchanged.
5. Frontend auth integration and edit/draft workflows
   - Wire the real account model into submissions and user sessions.
6. CRA to Vite migration
   - Move the frontend build once backend contracts and deployment assumptions are stable.

This order keeps backend contract changes ahead of the frontend bundler migration and avoids mixing three high-risk shifts in one PR stream.

## Suggested PR breakdown

1. `docs: audit current app and migration risks`
2. `design: define target data model and auth ownership rules`
3. `backend: add Amplify Gen 2 data model alongside migration plan`
4. `backend: migrate submissions and replace public api-key flow`
5. `frontend: integrate real auth and account-bound picks`
6. `frontend: add editable picks, lock timing, and draft saving`
7. `frontend: migrate CRA app to Vite`
8. `cleanup: remove Gen 1 artifacts and stale frontend dependencies`

## Unknowns to resolve before implementation

- Are historical submissions required to remain editable, viewable, or claimable by future users?
- Should one person be able to create multiple entries?
- Should public users still be able to view all picks and all entrant names after auth is introduced?
- Is the existing public AppSync endpoint still in active production use?
- Does the production domain need to remain at `/leaderboard/`?
